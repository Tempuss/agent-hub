## 🚀 5분 빠른 시작

### Step 1: 문제 정의 (5분)

**가장 중요한 단계** - 여기서 실패하면 모든 시간이 낭비됩니다.

```
Q1: 정확한 문제가 무엇인가? (증상 vs 근본 원인 구분)
Q2: 왜 이것이 문제인가? (비즈니스 영향도)
Q3: 성공은 무엇처럼 보이는가? (측정 기준)
```

**예시**:
- ❌ 문제: "우리 앱이 느리다"
- ✅ 문제: "사용자 온보딩 이후 이탈율이 8% (업계 평균 3%)"

### Step 2: 사고법 선택 (2분)

thinking-framework의 어떤 방법을 사용할지 선택:

| 상황 | 추천 사고법 |
|------|-----------|
| "왜 이런 일이 일어났나?" | 5 Why / Fishbone |
| "어떻게 혁신할 수 있나?" | SCAMPER / First Principles |
| "전략은 무엇인가?" | SWOT / GAP Analysis |
| "어떻게 개선할 수 있나?" | Pareto / PDCA |
| "빠르게 결정해야 한다" | OODA Loop |

### Step 3: 사고법 적용 (10-30분)

선택한 사고법의 단계별 프로세스를 따릅니다.
(각 방법의 상세 가이드는 `reference/` 디렉토리 참조)

**예시 - Pareto 분석**:
```
1. 데이터 수집: 고객 이탈 원인별 분류
2. 순위 정렬: 영향도 내림차순
3. 80/20 규칙 적용: 상위 20%가 80%의 이탈을 설명
4. 우선순위 결정: 상위 2-3개 원인에 집중
```

### Step 4: 연구 계획 (선택사항, 5-15분)

만약 신뢰도를 높이고 싶다면, 증거를 수집합니다:

```
Q1: 어떤 정보가 필요한가?
Q2: 신뢰할 수 있는 출처는 어디인가?
Q3: 얼마나 시간을 들일 수 있는가?
```

**연구 출처 등급**:
- **Tier 1 (신뢰도 90%)**: 공식 문서, 학술 논문, 정부 자료
- **Tier 2 (신뢰도 70%)**: 업계 리포트, 전문가 기사, 신뢰할 수 있는 매체
- **Tier 3 (신뢰도 50%)**: 커뮤니티 자료, 사용자 경험담
- **Tier 4 (신뢰도 30%)**: 소셜 미디어, 검증되지 않은 출처

### Step 5: 신뢰도 계산 (5분)

```
신뢰도 = 방법 신뢰도 × 증거 신뢰도

예시:
- Pareto 분석 정확도: 85% (잘 증명된 방법)
- 우리가 수집한 증거 신뢰도: 75% (Tier 1-2 출처 혼합)
- 최종 신뢰도: 85% × 75% = 64%

해석:
- 60% 이상 = 충분히 신뢰 가능한 결정
- 70% 이상 = 높은 신뢰 가능 결정
- 80% 이상 = 매우 높은 신뢰도
```

### Step 6: 실행 (계속 진행)

```
신뢰도 64%로 상위 2개 이탈 원인 개선에 집중
→ A/B 테스트로 실제 효과 측정
→ 결과로 다음 사고 사이클 개선
```

## 💡 실제 사용 사례

### Case 1: 스타트업의 자금 부족 위기

**상황**: 3개월 남은 런웨이, 경로가 불명확

**Step 1 - 문제 정의**:
```
문제: 현금 부족이 아니라 "명확한 매출 경로의 부재"
성공 기준: 3개월 내 MRR $50K 달성
```

**Step 2 - 사고법 선택**: SWOT + GAP Analysis

**Step 3 - SWOT 분석**:
```
강점: 탁월한 기술, 열정적인 팀
약점: 판매 프로세스 부재, 높은 번율
기회: 시장 연 40% 성장
위협: 자금이 풍부한 경쟁사
```

**Step 4 - 연구 (선택사항)**:
```
Q: 우리 시장 세그먼트의 일반적 판매 주기는?
Q: 경쟁사들의 GTM 전략은?
Q: 파트너십 기회는?

출처 찾기: 산업 리포트, 경쟁사 분석, 업계 컨퍼런스
```

**Step 5 - 신뢰도 계산**:
```
- SWOT 방법 신뢰도: 75% (잘 구조화되었지만 주관적)
- 우리 수집 증거: 80% (Tier 1-2 출처, 현장 경험)
- 최종 신뢰도: 75% × 80% = 60%

결론: 충분히 신뢰할 수 있는 전략
```

**Step 6 - 실행 계획**:
```
1. 파트너십 5개 파일럿 (60일)
2. 브릿지 펀딩 모금 (45일)
3. 내부 판매팀 구축 (90일)
```

### Case 2: 고객 이탈 문제 해결

**상황**: 이탈율 8% (업계 평균 3%)

**Step 1 - 문제 정의**:
```
문제: 온보딩 후 1주일 내 이탈 집중
성공 기준: 이탈율을 5% 이하로 감소
```

**Step 2 - 사고법 선택**: Pareto 분석

**Step 3 - 데이터 분석**:
```
원인별 이탈:
- 온보딩 어려움: 450건 (45%)
- 기능 부족: 280건 (28%)
- 고객 지원 미흡: 120건 (12%)
- 기타: 150건 (15%)

80/20 적용:
상위 2개 = 73%의 이탈
→ 온보딩 + 기능 개선에 집중
```

**Step 4 - 연구**:
```
Q: 다른 회사들의 온보딩 개선 성과는?
Q: 업계 벤치마크는?

출처: 케이스 스터디, SaaS 분석 리포트
```

**Step 5 - 신뢰도**:
```
- Pareto 정확도: 90% (데이터 기반)
- 증거 신뢰도: 85% (다수의 Tier 1-2 사례)
- 최종 신뢰도: 90% × 85% = 76.5%

높은 신뢰도 → 자신감 있게 투자 가능
```

**Step 6 - 실행**:
```
온보딩 개선:
- 비디오 튜토리얼 추가 (2주)
- 라이브 온보딩 세션 (1주일)
- 예제 프로젝트 제공 (3주)

→ 3주 후 측정 → 결과 기반 다음 단계
```

## 📊 신뢰도 점수 계산

### 계산 공식

```
신뢰도 = 방법 신뢰도 × 증거 신뢰도 × 상황 적합도
```

### 각 요소별 점수 방법

#### 1️⃣ 방법 신뢰도 (Method Credibility)

| 사고법 | 신뢰도 | 이유 |
|------|-------|------|
| 5 Why / Fishbone | 90% | 매우 체계적, 수십 년 증명됨 |
| SWOT | 75% | 잘 알려짐, 하지만 주관적일 수 있음 |
| Pareto | 90% | 데이터 기반, 통계적 근거 |
| SCAMPER | 80% | 창의성을 자극하지만 산발적 |
| First Principles | 85% | 논리적이지만 시간 소모 |
| OODA Loop | 85% | 동적 환경에 증명됨 |

#### 2️⃣ 증거 신뢰도 (Evidence Quality)

**출처별 점수**:
```
Tier 1 (학술/공식): 90%
- 학술 논문
- 정부 자료
- 회사 공식 발표

Tier 2 (전문가/신뢰 매체): 70%
- 업계 리포트
- 저명한 전문가 기사
- 신뢰할 수 있는 신문/매체

Tier 3 (커뮤니티): 50%
- 사용자 포럼
- 블로그 게시물
- 업계 컨퍼런스 발표

Tier 4 (소셜/미검증): 20%
- 트위터/소셜 미디어
- 댓글 섹션
- 검증되지 않은 주장
```

**여러 출처의 평균**:
```
예시: 3개 출처 수집
- 학술 논문: 90%
- 업계 리포트: 70%
- 전문가 블로그: 70%

평균 증거 신뢰도 = (90 + 70 + 70) / 3 = 76.7%
```

#### 3️⃣ 상황 적합도 (Contextual Fit)

```
100% = 모든 조건 충족
- 시간 충분
- 리소스 충분
- 문제 유형이 사고법과 정확히 일치

70% = 일부 조건 미충족
- 시간 부족하지만 수행 가능
- 리소스 제약 있음
- 문제가 대체로 맞음

40% = 여러 조건 미충족
- 시간 부족
- 리소스 최소한
- 문제가 부분적으로만 맞음
```

### 신뢰도 해석

```
최종 신뢰도 점수:

80% 이상: 매우 높은 신뢰도
- 자신감 있게 major decision 실행
- 장기 전략에 사용
- 큰 리소스 투입 정당화

60-79%: 중간-높은 신뢰도
- 합리적인 결정 가능
- MVP 또는 파일럿으로 검증
- 점진적 실행

40-59%: 중간 신뢰도
- 방향성은 맞지만 불확실성 있음
- 작은 규모로 테스트
- 정보 부족 영역 파악 후 연구 추가

0-39%: 낮은 신뢰도
- 결정 연기 권장
- 더 많은 연구/정보 필요
- 전문가 상담 고려
```

## 🔄 통합 워크플로우

### 사고법 + 연구 통합 프로세스

```mermaid
1. 문제 정의 (30-60분)
   ↓
2. 사고법 선택 및 적용 (20-60분)
   ↓
3. 초기 분석 결과 도출
   ↓
4. [신뢰도 평가]
   ├─ 80% 이상? → Step 6 (즉시 실행)
   ├─ 60-79%? → Step 5 (조사 추가)
   └─ 60% 미만? → Step 5 (광범위 연구)
   ↓
5. [필요 시] 연구 계획 및 실행 (1-7일)
   - 핵심 가정 검증
   - 벤치마크 데이터 수집
   - 사례 연구 분석
   ↓
6. 신뢰도 재계산
   ↓
7. 실행 계획 및 의사결정
   ↓
8. 결과 측정 및 학습
```

### 시간대별 패턴

**패턴 1: 빠른 의사결정 (1-2시간)**
```
적합한 경우: 분기별 기능 우선순위, 작은 결정
- 문제 정의: 10분
- 사고법 적용: 30분
- 신뢰도 평가: 5분
- 결정: 15분
- 연구: 최소화 (내부 데이터만 사용)
```

**패턴 2: 전략적 의사결정 (1-3일)**
```
적합한 경우: 새로운 시장 진입, 사업 전략 수립
- 문제 정의: 60분
- 사고법 적용 (SWOT+GAP): 2시간
- 연구 (targeted): 6-12시간
- 신뢰도 계산: 30분
- 최종 의사결정: 1시간
```

**패턴 3: 혁신/문제 해결 (3-7일)**
```
적합한 경우: 신제품 개발, 주요 문제 해결
- 문제 정의: 2시간
- 사고법 적용 (Design Thinking + First Principles): 8시간
- 연구 (포괄적): 16-24시간
- 사용자 연구: 8-16시간
- 신뢰도 계산: 1시간
- 정제 및 계획: 2시간
```

## ⚠️ 흔한 실수와 해결책

### 실수 1: 문제 정의 스킵하기

❌ **증상**:
```
"이건 명확한 문제야. 지금 당장 분석해"
→ 완벽하게 분석했는데 문제가 아니었음
```

✅ **해결책**:
```
문제 정의에 30-60분을 투자
- 증상 vs 근본 원인 구분
- 이해관계자 인터뷰
- 성공 기준 정의
```

### 실수 2: 오소드 신뢰도만 신뢰하기

❌ **증상**:
```
내부 데이터만 분석 (신뢰도 40%)
→ 현실과 맞지 않는 결정
```

✅ **해결책**:
```
Tier 1-2 외부 증거 추가
- 업계 벤치마크
- 경쟁사 사례
- 전문가 자료
```

### 실수 3: 사고법-문제 유형 불일치

❌ **증상**:
```
혁신 문제에 DMAIC 적용 (0% 성공률)
→ 창의성 억압, 결과 없음
```

✅ **해결책**:
```
문제 유형과 사고법 일치:
- 혁신 → SCAMPER/First Principles
- 개선 → PDCA/Pareto
- 분석 → 5Why/Fishbone
```

### 실수 4: 신뢰도 계산 스킵하기

❌ **증상**:
```
"분석했으니 이제 실행"
→ 잘못된 가정으로 시간/비용 낭비
```

✅ **해결책**:
```
신뢰도 점수로 실행 규모 결정:
- 80% 이상: 전사 규모 실행
- 60-79%: MVP/파일럿
- 60% 미만: 추가 연구 필요
```

## 🎯 빠른 참조 가이드

### 의사결정 트리

```
문제가 있는가?
│
├─ 명확한 문제가 아닌가?
│  └─ → 문제 정의 먼저!
│
├─ "왜?"라고 묻고 싶은가?
│  └─ → 5 Why / Fishbone
│
├─ "어떻게 다를 수 있나?"
│  └─ → SCAMPER / First Principles
│
├─ "전략은?"
│  └─ → SWOT + GAP Analysis
│
├─ "어떻게 개선하나?"
│  └─ → Pareto / PDCA
│
└─ "빠르게 결정?"
   └─ → OODA Loop
```

### 시간별 투자 가이드

| 의사결정 중요도 | 추천 시간 | 권장 방법 |
|-------------|---------|---------|
| 낮음 (작은 최적화) | 30분 | 빠른 분석만 |
| 중간 (분기별 전략) | 2-4시간 | 사고법 + 내부 데이터 |
| 높음 (사업 방향) | 1-2일 | 사고법 + 조사 |
| 매우 높음 (장기 전략) | 3-7일 | 사고법 + 광범위 연구 |
